核心定位

iFLOW CLI 的多智能体工作流扩展插件，通过角色化的智能体架构和结构化的工作流管道，帮助开发者更高效地进行代码编写、调试、规划和验证。

⚡

核心能力

  • 多智能体协作编排
  • 结构化工作流管道
  • LSP 语言服务集成
  • 自动格式化支持
  • 原生钩子控制
🔗

技术栈

  • TypeScript/Bun (cli-lsp-client)
  • Python (脚本和工具)
  • Shell (格式化插件)
  • TOML (命令配置)
  • Markdown (智能体定义)

优化建议

识别的问题与改进方案

🔴 状态持久化路径不一致

高优先级

📋 当前问题

状态文件分散在两个不同路径：

  • .iflow/state/ - mode.json, approval.json, reasoning.json
  • .omi/state/ - hud.json

这导致配置管理混乱，难以统一维护。

🛠️ 优化方案

  • 统一状态存储路径为 .omi/state/
  • 创建状态管理抽象层
  • 添加状态迁移脚本
  • 更新所有技能的状态路径引用
// 统一状态路径常量 const STATE_DIR = '.omi/state'; const STATE_FILES = { mode: `${STATE_DIR}/mode.json`, approval: `${STATE_DIR}/approval.json`, reasoning: `${STATE_DIR}/reasoning.json`, hud: `${STATE_DIR}/hud.json` };

✨ 预期效果

  • 状态管理一致性提升 100%
  • 减少配置错误风险
  • 简化备份和迁移操作
  • 提升代码可维护性
💡 改进后

所有状态文件集中在 .omi/state/ 目录，通过统一 API 访问，支持原子操作和事务。

🟡 循环限制配置分散

中优先级

📋 当前问题

不同技能的循环限制硬编码在各自的 SKILL.md 中：

  • loop: max 3 cycles
  • autopilot: max 5 cycles
  • ralph: 无限制直到通过

缺乏全局配置机制，难以根据场景调整。

🛠️ 优化方案

  • 创建全局循环配置文件 .omi/config.json
  • 支持按模式自动调整循环限制
  • 添加运行时覆盖能力
  • 实现循环计数器持久化
// .omi/config.json { "loopLimits": { "default": 5, "speed": 3, "deep": 10, "autopilot": 5, "ralph": -1 } }

✨ 预期效果

  • 配置集中管理，易于调整
  • 支持场景化配置预设
  • 避免无限循环风险
  • 提供更好的用户体验控制
